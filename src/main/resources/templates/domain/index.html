How does this work?

Domains, Tags, and Documents

  • Domain is the top-level organizer. It represents a named theme or knowledge bucket like Brewing, Cookbooks, or Permaculture.
  • Document belongs to one domain and stores source metadata: title, author, filename/path, source type, and ingestion state. Documents are the units you ingest and later search over.
  • Tag is a domain-scoped label hierarchy. Tags are scoped to a single domain and can be expressed as hierarchical paths like cuisine.italian or technique.fermentation.
  • Documents are associated with tags, but tags are not global—they live within the domain’s namespace.

How they work together

  • A domain groups documents by theme.
  • A document is a source file inside that domain.
  • Tags classify documents within the domain and enable filtered retrieval.
  • During ingestion, each document is split into Chunks, and chunks carry denormalized metadata:
    • domainId so filtered vector search can work without joins
    • tagPaths so chunk retrieval can respect tag filters

Why this model

  • the domain simple and scalable
  • Supports filtered vector search by domain + tag paths
  • Keeps domain and document metadata lightweight
  • Allows the retrieval/query layer to build nested views only when needed
